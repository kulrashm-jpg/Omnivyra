# CMO Portfolio Radar View

Cross-campaign intelligence for company-level health. **Visibility: COMPANY_ADMIN only.** Execution remains campaign-scoped; no cross-campaign execution board.

---

## Role-based visibility

| Rule | Implementation |
|------|----------------|
| Who can see Portfolio | `userRole === 'COMPANY_ADMIN'` only |
| Toggle | "Portfolio" center-view button is rendered only when `isCompanyAdmin` in `EnterpriseExecutionLayout` |
| Stored preference | If stored view is `portfolio` and user is not COMPANY_ADMIN, initial view falls back to role-based default (Radar/Pipeline) |

---

## Layout (three sections)

1. **Company AI Narrative**  
   Overall execution health summary and campaigns needing attention. Generated by `getCompanyPortfolioHealth` from aggregated health (overdue, blocked, pending, unassigned).

2. **Campaign Health Grid**  
   One card per campaign:
   - Campaign name  
   - **Risk Now** (primary): score 0–100 with level color (Healthy / Watch / Critical)  
   - **Predicted** (secondary): next-week score with trend ↑ (increasing) / → (stable) / ↓ (improving)  
   - One-line explanation (e.g. "Risk increasing due to approval delays.")  
   - Health color, overdue count, pending approvals, stage bottleneck  
   - **Top risk contributors** (short explainable line)  
   - **Sort by risk** (default: highest first) or by name (A–Z)

3. **Cross-Campaign Attention Feed**  
   Highest-priority activities across all campaigns (overdue → blocked → waiting_approval → unassigned), limited to top N (e.g. 15). Each row shows activity title and campaign name; click opens that campaign’s Radar and optionally the activity.

---

## Behavior

- **Click campaign card** → Select that campaign (`onSelectCampaign(campaignId)`), switch center view to **Radar**, close or open side panel depending on whether an activity was selected.
- **Click attention feed row** → Select that campaign and activity (`onSelectCampaign(campaignId, activityId)`), switch to **Radar**, open side panel with that activity.
- **No cross-campaign execution.** All execution (Pipeline, Radar, activity panel) is scoped to the selected campaign.

## Architecture

### Portfolio aggregation (`lib/campaign-health-engine.ts`)

- **Reuse:** `getCampaignHealth(campaignId, fetchActivities, options)` per campaign.
- **New:** `getCompanyPortfolioHealth(campaigns, fetchActivities, options)`:
  - `campaigns`: `Array<{ id: string; name: string }>` (e.g. from `CampaignContextItem[]`).
  - `fetchActivities`: `(campaignId: string) => Promise<Activity[]>` (injected; e.g. API or parent state).
  - Runs `getCampaignHealth` for each campaign in parallel, then:
    - **Company narrative:** Rule-based summary (e.g. “X items past due or blocked across N campaigns”, “Consider reviewing: A, B, C”).
    - **Campaign cards:** Per-campaign `CampaignHealthCard` (health color from overdue/blocked → red; pending/unassigned → orange; else green; bottleneck from `stageHealthSummary.hasIssues`).
    - **Attention feed:** Merge all `attentionItems`, sort by priority, slice to top N; each item includes `campaignId`, `campaignName`, `activityId`, `activityTitle`, `reason`, `priority`.

### Campaign Risk Score (0–100, rule-based, explainable)

- **Goal:** Quick executive prioritization per campaign. No black-box AI.
- **Formula:** Weighted sum of signals, capped at 100.
  - **High weight:** overdue (+25 each), blocked (+25 each).
  - **Medium weight:** pending approval (+10 each), unassigned (+10 each), stage bottleneck (+15 per stage with issues).
  - `score = min(100, round(overdue×25 + blocked×25 + pending×10 + unassigned×10 + bottleneckStages×15))`.
- **Levels:** 0–30 → Healthy (green), 31–60 → Watch (orange), 61–100 → Critical (red).
- **Top contributors:** List of up to 5 contributing signals (e.g. "2 overdue", "1 blocked", "3 pending approval") sorted by points contributed, for short UI explanation.
- **Implementation:** `computeCampaignRiskScore(health)` in `lib/campaign-health-engine.ts`; `RISK_WEIGHTS` exported for transparency. Each `CampaignHealthCard` includes `riskScore`, `riskLevel`, `riskContributors`.
- **Recalculation:** Risk is derived from `CampaignHealth`; whenever portfolio is loaded (e.g. on opening Portfolio view or when `campaigns` / `fetchActivitiesForCampaign` change), scores are recomputed.

### Predicted Risk (Next Week)

- **Goal:** Proactive early-warning; rule-based trend from current health (no time-series required).
- **Service:** `getCampaignRiskPrediction(campaignId, fetchActivities, options)` and pure `computeCampaignRiskPrediction(health)` in `lib/campaign-health-engine.ts`.
- **Output:** `CampaignRiskPrediction`: `predictedScore` (0–100), `trend` ('increasing' | 'stable' | 'improving'), `explanation` (one line).
- **Trend rules (rule-based):**
  - **Approval backlog trend:** pendingApprovalCount ≥ 2 → factor for "increasing"; explanation "Risk increasing due to approval delays." when dominant.
  - **Overdue trend:** overdueCount > 0 → "increasing"; "overdue backlog".
  - **Stage congestion:** bottleneck stages + multiple activities → "increasing"; "stage congestion".
  - **Unassigned:** unassignedCount ≥ 2 → factor for "increasing"; "unassigned work".
  - **Improving:** current score ≤ 30 and no overdue/blocked → "improving"; "maintain current pace."
  - **Stable:** otherwise; "monitor key metrics."
- **Predicted score:** increasing → min(100, current + 12); stable → current; improving → max(0, current − 8).
- **Update frequency:** Daily or throttled; portfolio load already acts as refresh (no extra real-time updates).
- **UI:** Current risk remains primary; prediction shown as secondary (Risk Now: X | Predicted: Y ↑/↓/→ + explanation).

### Suggested Options (user-choice model)

- **Goal:** AI offers 2–3 alternative options; user decides which path to take. AI never auto-executes.
- **Section title:** "Suggested Options". Each option: title, short reason (1 line), impact level (low/medium/high), [Open Related Items].
- **Strategies:** Clear blockers (CLEAR), assign ownership (ASSIGN), advance workflow (ADVANCE). Options do not duplicate each other.
- **Service:** `computePreventiveActions(health, prediction)` and `getPreventiveActions(campaignId, fetchActivities)` in `lib/campaign-health-engine.ts`.
- **Rules:** User explicitly selects an option; selecting opens campaign radar with relevant filter. No automatic workflow changes.
- **Action shape:** `PreventiveAction`: category, label (title), reason?, impactLevel, suggestedFilter? (stage / approvalStatus for “Open Related Items”).
- **UI:** Section "Suggested Options"; each option shows title, reason, impact badge, [Open Related Items]; clicking opens campaign radar with suggested filter applied (layout calls `onFiltersChange(suggestedFilter)` when opening).
- **Integration:** Each `CampaignHealthCard` includes `preventiveActions`; populated in `getCompanyPortfolioHealth`.
- **Adaptive learning (invisible):** System learns which option types the user prefers and reorders future suggestions silently. No leadership-pattern insights or behavioral analytics UI. Data tracked per user: user_id, selected_option_type, timestamp, campaign_id (optional). Frequency + recency weighting. Effect: reorder preventive action options only; all options remain visible; no explicit explanation of reordering. Order is frozen for the current view; apply new ordering on next load or refresh only. See `docs/AI-ADAPTATION-INVISIBLE.md`.

### Types (engine)

- `CampaignHealthCard`: … riskScore, riskLevel, riskContributors, prediction, **preventiveActions** (PreventiveAction[]).
- `CampaignRiskScore`: score, level, contributors. `RiskLevel`: 'healthy' | 'watch' | 'critical'. `RISK_WEIGHTS`: overdue, blocked, pendingApproval, unassigned, bottleneckPerStage.
- `CampaignRiskPrediction`: predictedScore, trend ('increasing' | 'stable' | 'improving'), explanation. `RiskTrend` = trend type.
- `PreventiveAction`: category, label (title), reason?, impactLevel (low|medium|high), suggestedFilter?. `ImpactLevel` = 'low'|'medium'|'high'. `PreventiveActionFilterHint`: stage?, approvalStatus?.
- `PortfolioAttentionItem`: campaignId, campaignName, activityId, activityTitle, reason, priority.
- `CompanyPortfolioHealth`: companyNarrative, campaignsNeedingAttention, campaignCards, attentionFeed.

### Component structure

- **`CmoPortfolioRadarView`** (`components/execution-layout/CmoPortfolioRadarView.tsx`):
  - Props: `portfolio: CompanyPortfolioHealth | null`, `loading: boolean`, `onSelectCampaign(campaignId, activityId?, suggestedFilter?)`.
  - Renders: Company AI Narrative, Campaign Health Grid (cards with Risk Now, Predicted, **Suggested Options** with title/reason/impact + [Open Related Items]), Cross-Campaign Attention Feed.
  - **onSelectCampaign(campaignId, activityId?, suggestedFilter?):** when suggestedFilter is provided (e.g. from Open Related Items), layout applies filter when opening campaign radar.
  - **Sort:** Campaign cards can be sorted by **Risk** (default: highest first) or **Name** (A–Z) via toolbar above the grid.
  - No data fetching; receives aggregated data from layout.

- **`EnterpriseExecutionLayout`**:
  - New prop: `fetchActivitiesForCampaign?: (campaignId: string) => Promise<Activity[]>`.
  - When `centerView === 'portfolio'`: loads portfolio via `getCompanyPortfolioHealth(campaigns, fetchActivitiesForCampaign)`, renders `CmoPortfolioRadarView`, passes `handlePortfolioSelectCampaign` (calls `onSelectCampaign(campaignId)`, `setCenterView('radar')`, sets selected activity and panel mode if `activityId` provided).
  - When entering portfolio view, activity panel is closed (execution stays campaign-scoped).

### Data flow

- Parent supplies `campaigns` and optionally `fetchActivitiesForCampaign`.
- Layout calls `getCompanyPortfolioHealth` when Portfolio tab is active; results stored in `portfolioData` and passed to `CmoPortfolioRadarView`.
- If `fetchActivitiesForCampaign` is omitted, a no-op that returns `[]` is used so portfolio still renders (empty health per campaign).
